二、第一优先级：Panel v2，不要只做二元 AND/OR/XOR

你现在的 panel 方向已经有 7 个 solved，而且 near-miss 最大类是 panel_rule_near_miss，说明这里仍然是最大金矿。之前的 PanelSeparatorBinaryOpRule 有效，但它只覆盖了“两个 panel 做二元逻辑运算”。现在要扩成：

PanelSemanticRule

它不只是比较两个 panel，而是枚举下面这些输出来源：

1. output = 某一个 panel
2. output = 某一个 panel 去掉背景/分隔线后的 crop
3. output = 含有唯一颜色的 panel
4. output = 非背景面积最大/最小的 panel
5. output = 与其他 panel 不同的那个 panel
6. output = 多个 panel 的 union/intersection/majority
7. output = 某 panel 经过 color map
8. output = 某 panel 经过 mirror/rotate
9. output = 某 panel 的 bbox crop
10. output = 所有 panel 中缺失/多出来的部分

你现在的 PanelSelectByColorRule 真实 probe 是 0，一个原因是它太保守：要求共享 output shape、共享 panel layout、同一 panel index。这个限制应该放松。改成：

每个 train case 可以有不同 panel 尺寸和不同 layout；
但是选择语义必须一致。

例如：

case1 选“含唯一颜色的 panel”
case2 也选“含唯一颜色的 panel”
case3 也选“含唯一颜色的 panel”

不要求它们都是第 2 个 panel，也不要求 panel 尺寸一样。

给 Codex 的实现要求

让它新增：

PanelSemanticRule

Python probe 先实现这些 selector：

select_panel_by_unique_color
select_panel_by_color_absent_from_others
select_panel_by_most_non_background
select_panel_by_least_non_background
select_panel_by_different_shape
select_panel_by_matching_output_after_colormap
select_panel_by_matching_output_after_crop
select_panel_by_matching_output_after_rotate_or_mirror

Panel layout 枚举不要只看完整 separator，还要支持：

等宽 panel
等高 panel
由全背景行/列分隔
由某颜色线分隔
由 repeated separator 分隔
2x2 panel
1x3 / 3x1 panel
不规则 panel，只要可以由空行/空列分割

ONNX builder 先只提交能安全编译的：

1. 固定 remap panel crop
2. panel + color map
3. panel + rotate/mirror
4. panel bbox crop

如果 selector 是动态的但 ONNX 暂时难写，就只放进 probe 报告，不进 submission。

预期收益：+8 到 +18 个。

三、第二优先级：DynamicBBoxCropRule，这是冲 80 的关键

你现在的 ObjectSelectionRule 太保守，只支持 train 中 selected bbox 完全静态一致，所以没有新增有效 solved。日志里也写了：要继续冲 solved，应扩展 largest/smallest component crop、crop object then normalize to top-left、crop by marker color、crop interior of frame 等 object/crop 语义。

下一步应该做：

DynamicBBoxCropRule
ObjectNormalizeRule
FrameInteriorCropRule
MarkerGuidedCropRule

这些规则的本质是：

在输入里找一个对象 / bbox / frame / marker 区域
然后把它裁剪出来，放到输出左上角
1. DynamicBBoxCropRule

Python probe 枚举：

bbox of all non-background
bbox of each color
bbox of largest component
bbox of smallest component
bbox of unique-color component
bbox of component touching marker
bbox of component not touching border
bbox of object inside frame

输出匹配形式：

output = bbox crop
output = bbox crop + color map
output = bbox mask
output = bbox frame
output = bbox interior
output = normalized object at top-left
2. ONNX builder 的关键

不要一开始做通用 connected components 的 ONNX，这很难。先做这些可以安全编译的情况：

color-specific bbox
non-background bbox
frame-color bbox
marker-color bbox

这些可以用 ONNX 允许算子动态算：

mask = selected color / non-background
row_any = ReduceSum(mask over width) > 0
col_any = ReduceSum(mask over height) > 0
top = ArgMax(row_any)
left = ArgMax(col_any)
bottom = last active row
right = last active col

ArgMax 不在题目禁用列表里；但必须让 Codex 单独验证 onnx.checker、onnxruntime 和你的 inspect_submission。题面禁用的是 Loop / Scan / NonZero / Unique / Script / Function，所以这类动态 bbox 可以尝试，但一定要通过你的约束检查。

如果动态 Gather 太麻烦，先做静态展开：

枚举 top,left in 0..29
为每个 top,left 生成 candidate crop
根据 mask 判断哪个 candidate 有效
用 selector 选择

虽然图大一点，但 30×30 以内仍可能低于 1.44MB。

预期收益：+10 到 +20 个。

这是最可能把你从 45 推到 60+ 的方向。

四、第三优先级：Frame / Interior / Border 规则

很多 ARC 任务不是“对象选择”，而是：

找一个矩形框
取框内内容
填框内部
擦掉框
保留边框
把框内物体移出

新增：

FrameInteriorRule
FrameExtractRule
FrameRemoveRule
FrameFillRule

Python probe：

1. 枚举每种颜色是否构成矩形 frame。
2. 找 frame 的 top/bottom/left/right。
3. 测试 output 是否等于：
   - frame 内部 crop
   - frame 外部删除后的 grid
   - 只保留 frame 内对象
   - frame 内部填某颜色
   - frame 边框换色
   - frame 内对象 normalize 到左上角

ONNX builder：

先支持：

静态 frame 位置
color-specific dynamic frame bbox
frame interior crop
frame fill mask

预期收益：+4 到 +10 个。

这个和 DynamicBBoxCropRule 可以共享 bbox compiler。

五、第四优先级：ObjectEditRule，专攻 same-size 失败任务

你的 same-size 失败任务仍然很多。之前 LocalFill、HoleFill 已经吃掉一部分，但还不够。下一步 same-size 方向不要继续做单纯 3×3 local rule，而要做 object-level edit：

ObjectOutlineRule
ObjectFrameRule
ObjectRemoveNoiseRule
ObjectColorRoleRule
ObjectMoveTowardMarkerRule
ObjectCompleteSymmetryRule
1. ObjectOutlineRule / ObjectFrameRule

匹配：

输出 = 给对象画外轮廓
输出 = 给对象画 bbox frame
输出 = 对象内部填色
输出 = 对象边界换色

Python probe：

for each component:
    compute bbox
    compute perimeter / interior / border-adjacent cells
    test output

ONNX 初版可以只支持：

基于颜色 mask 的 dilation - mask 得到 outline
bbox frame 如果 bbox 可动态求
2. RemoveNoiseRule

匹配：

删除孤立点
删除面积最小对象
删除某颜色噪声
保留最大对象
保留重复出现的主图案

Python probe 非常容易，ONNX 先保守：

只支持局部孤立点删除
只支持 3x3 neighbor count
只支持固定小组件删除
3. ObjectMoveTowardMarkerRule

匹配：

某颜色对象朝 marker 移动
某对象复制到 marker 位置
某对象沿 marker 指示方向移动

这类 ARC 也常见，但 ONNX 稍复杂。先做 probe，不急着 build。

预期收益：+5 到 +15 个。

六、第五优先级：Rule Composition，不是新增单条规则

到 45 之后，很多任务不再是单一变换，而是两步组合：

先 crop，再 color map
先 select panel，再 rotate
先 extract object，再 normalize
先 mirror completion，再 recolor
先 tile，再 mask

你现在每条规则大多是单步规则。要冲 80，必须加一个轻量组合搜索：

ComposedRuleSearch

不要做大规模 DSL，先做两层组合就够：

Transform A -> Transform B

可组合的 A：

crop
panel_extract
bbox_extract
object_mask
mirror
rotate
scale
tile

可组合的 B：

color_map
recolor_non_background
normalize_to_top_left
mirror
rotate
crop_background
overlay

Python probe：

for rule_a in simple_extractors:
    mid = rule_a(input)
    for rule_b in simple_finishers:
        out = rule_b(mid)
        compare to expected

规则必须跨所有 train case 使用同一组合语义，但允许参数由 input 自动推断。

ONNX builder：

先只提交这些安全组合：

spatial_remap + color_map
spatial_remap + rotate/mirror
panel_extract + color_map
bbox_crop + color_map
bbox_crop + normalize

预期收益：+8 到 +20 个。

这一步是从 60 往 80 冲的核心。

七、不要再依赖 first_version_rules() 的固定手写顺序

现在规则越来越多，如果仍然按固定顺序逐个 match，可能会发生：

一个粗规则先 match，但 cost 高或泛化差；
后面的更好规则没机会成为 best。

建议改成：

每个 task 允许所有规则都生成 candidate
最后统一 validate + cost 排序

也就是：

Rule.match -> Candidate
所有 Candidate 都进入候选池
validate 全部候选
选择 validation_passed 且 cost 最低者

对于 probe-only 的规则，只进入 report，不进入 candidate pool。

这能避免新规则被旧规则遮挡。

八、增加一个 candidate_discovery_report.csv

你目前有 probe_summary.csv 和 rule_near_miss.csv，但还缺一个能指导冲 80 的报告：

outputs/reports/candidate_discovery_report.csv

每个 failed task 记录：

task_id
candidate_rule
python_transform_passed
onnx_builder_available
onnx_validation_passed
blocked_reason
estimated_codegen_difficulty
expected_gain_bucket

其中 blocked_reason 要细：

builder_missing_dynamic_bbox
builder_missing_dynamic_panel_select
requires_component_normalization
requires_panel_variable_layout
requires_composed_rule
requires_object_motion
requires_counting

这可以避免 Codex 继续盲目写低收益规则。
